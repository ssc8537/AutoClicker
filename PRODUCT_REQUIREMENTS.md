# 自动连招：唯一产品需求

本文件保存当前仍然有效的全部真实需求。它不是历史日志，也不是阶段计划；未来 AI 只按这里和 `PROJECT_ROADMAP.md` 工作，不从旧阶段文档猜需求。

## 1. 产品目标

- 面向 Windows 11 的个人游戏自动连招工具，中文界面，程序名称“自动连招”。
- 用户用可信本地 Python 文件描述连招，入口固定为 `run(player)`。
- 软件必须适合小白：清楚、漂亮、操作少；每次开发先自动检查，再提供简短中文验收。
- 优秀案例 1 只用于核对单键触发、快捷连点、紧凑 UI 和输入安全；不复制其源码、品牌、资源或 JSON/QIM 格式。

## 2. 页面与桌面体验

- 四个页面：宏库、触发、功能、设置。
- 宏库：新建、编辑、静态校验、保存、改名、删除、AI 提示词；错误宏不执行，用户注释保留。
- 触发：每个宏可直接修改启用状态、单一触发键、按下/切换、次数和速度，保存后立即生效；状态颜色不能被悬停覆盖。
- 功能：快捷连点必须保持一排；共享游戏键位位于快捷连点下方，名称和物理键都可修改。
- 设置：自定义全局启停键、OSD 开关、提示音开关；不放共享游戏键位。
- OSD 必须按完整提示文字和脚本名称自动扩宽；超过可用屏幕宽度时换行增高，不得从左右裁掉名称，并且不能抢游戏焦点或拦截鼠标。
- 窗口、任务栏、EXE 和托盘统一使用用户角色图标；关闭、最小化到任务栏、隐藏到托盘是三个不同动作。
- 程序为无黑色命令窗口的文件夹式便携 EXE；`config/`、`macros/`、`logs/` 位于 EXE 同级并可写。

## 3. 输入、触发与安全

- 每次启动后全局自动化默认启用，设置页立即显示“热键已启用”；启动本身不运行宏，只有真实按下已启用宏的触发键才执行。第一次按当前全局键切换为禁用，再按一次恢复启用。
- 全局键、宏触发键和快捷连点键均支持一个标准键盘键或五鼠标键；不支持两个键组合触发一个宏。
- `down`：物理按下启动，物理松开立即停止；`switch`：第一次物理按下启动，第二次物理按下停止，松开无动作。
- 同一触发键可以绑定多个宏并发运行，每个宏独立停止。
- 自己发送的输入使用 `MAPL` 标记并由监听层过滤，不能再次触发宏、快捷连点、全局开关或提示音。
- 正式 Windows EXE 请求管理员权限，与可能以管理员运行的游戏保持相同权限层级；`SendInput` 返回失败时必须限频写入诊断日志。
- 关闭程序时停止全部播放器和快捷连点、释放输入、注销 hook、关闭 OSD、等待线程，最终不得残留 `MyAutoPlayer.exe`。
- 只允许一个程序实例；第二次启动必须提示“自动连招已经在运行”。

## 4. Python 宏契约

每个宏必须包含：

```python
NAME = "宏名称"
HOTKEY = "mouse_back"
MODE = "down"
COUNT = 0
SPEED = 1.0
ENABLED = True

def run(player):
    player.战技()
```

- `HOTKEY`：一个支持的物理键；不能与当前全局键冲突。
- `MODE`：`down` 或 `switch`。
- `COUNT`：0–99；0 表示由程序重复运行到收到停止请求。
- `SPEED`：0.01–8.0，只缩放 `player.sleep()`。
- `ENABLED`：布尔值。

真实公开 API：

- `player.tap(key, hold_ms=20)`
- `player.sleep(ms)`
- `player.切换(1|2|3)`
- `player.战技()`、`player.声骸()`、`player.大招()`、`player.跳跃()`、`player.处决()`
- `player.按键("当前自定义动作名称", hold_ms=20)`
- `player.mouse_click(button, hold_ms=10)`
- `player.mouse_down(button)`、`player.mouse_up(button)`
- `player.mouse_repeat(count, button, interval_ms=10)`

用户确认的语义：角色 1 默认物理键为 `1`；固定中文函数按槽位工作。动作改名后，可用 `player.按键("改后的名称")`。正确函数名是 `player.战技()`，不是“战绩”。

## 5. 共享动作和连招作者规则

- 固定槽位：角色 1/2/3、战技、声骸、大招、跳跃、处决、扩展键 1/2/3。
- 所有显示名称和物理键可改；显示名称不能重复，物理键允许重复。
- 程序不识别当前角色、能量、资源、动画、技能冷却或游戏状态；连招必须使用用户提供的确定动作和毫秒时间线。
- 项目现有保守作者规则：普通切人前 50ms、快速回切 1080ms、大招后切人 1500ms；用户实测值优先。
- 当前新增角色宏使用：1 号秧秧、2 号琳奈、3 号千咲。旧 JSON 中“卡提”“夏空”“千笑”等名称不采用。

## 6. AI 提示词

- 程序内提示词必须让陌生 AI 只读后即可生成完整、可保存、可中断的 Python 宏。
- 内容覆盖全部真实 API、宏元数据、完整键值、当前全局键、当前共享动作、三角色资料模板、战斗逻辑、示例和安全禁区。
- 当前全局键和共享动作必须在打开窗口时动态读取，不能复制过期默认值。
- 角色事实缺失时外部 AI 只能询问最少必要问题，禁止编造技能、资源、冷却和动画。
- 提示词当前/默认文件改用 UTF-8 Markdown；阅读窗口渲染 Markdown，排版清楚。
- AI 提示词窗口必须非模态：可拖到旁边阅读，同时继续点击宏库、触发、功能和设置。

## 7. 视觉与性能

- 设置页提供两个互不绑定的外观选项：`经典粉红 / 樱空花园`主题，以及`经典顶部标签 / 画册侧边导航`布局；任意组合均可即时切换并保存。
- 经典粉红与经典顶部标签必须完整保留，并提供“一次点击恢复经典外观”的安全退路。
- 樱空花园从用户参考图提取樱花粉、天空蓝、薄荷绿、淡桃橙、云白和柔光卡片感；人物大图不铺在操作区，优先保证表格与控件可读性。
- 画册侧栏顶部使用用户提供素材裁出的圆形动漫头像；侧栏保持约 108px、头像和“宏库/触发/功能/设置”全部水平居中，不得挤占主要操作区。
- 宏库“序号”表头与数字必须居中；表格名称列随窗口宽度自动扩展。
- 无边框主窗口支持上、下、左、右和四个角拖动缩放；经典布局最小宽度 642px，画册布局最小宽度 760px，切换画册时自动适配为约 840px 且不得超出当前屏幕。
- 触发页和宏库的选中行使用淡蓝背景，但不得覆盖单元格原有文字色；启用保持绿，禁用/错误保持红，普通或紫色文字保持原色。
- 用户角色图标必须尽量填满图标画布，减少透明空白，但不能裁掉头发、发卡或脸部。
- 视觉源文件迁入 `my-automation-tool/assets/`，生成脚本从该稳定位置创建多尺寸 ICO；根临时图标包完成迁移后删除。
- 保持文件夹式 PyInstaller 构建，不退回启动更慢的单文件解包方案。
- 优化目标是减少可控的同步初始化、重复扫描和阻塞等待；不得为追求数字牺牲 hook ready、安全禁用或退出清理。

## 8. 已知问题与证据边界

- 中文输入法下字母输出会进入 IME 组合态；用户已接受英文输入法正确、中文偶发轻微问题，不再阻塞开发。
- 若未来重查中文侧键边沿，只能先记录真实 hook -> FIFO -> 状态机 -> stop 原始证据，再决定是否替换原生 hook；禁止猜改。
- OK-WW 是用户独立程序，不属于本项目残留进程。
- 游戏内无效果要先区分“触发链”和“发送链”：日志已记录宏启动则不得猜改 hook；管理员版仍无效时检查 `SendInput` 返回、前台窗口和游戏过滤。未取得证据前禁止加入或安装驱动输入。
- 外部 QuickInput JSON 只允许一次性、只读转换为本项目 Python 宏；点击、等待和固定循环可转换，图像/OCR、坐标和窗口条件必须明确标为不可等价转换，禁止偷偷改成无条件输入。

## 9. 永久不做

录制、鼠标坐标/轨迹、滚轮自动化、窗口绑定、定时运行、OCR/图像/颜色识别、角色状态判断、手柄、变量流程、按键/鼠标屏蔽、案例 JSON/QIM、两个键组合触发、联网技能数据库和自动更新。
