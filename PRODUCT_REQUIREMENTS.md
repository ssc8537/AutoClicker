# 自动连招：唯一产品需求

本文件保存当前仍然有效的全部真实需求。它不是历史日志，也不是阶段计划；未来 AI 只按这里和 `PROJECT_ROADMAP.md` 工作，不从旧阶段文档猜需求。

## 1. 产品目标

- 面向 Windows 11 的个人游戏自动连招工具，中文界面，程序名称“自动连招”。
- 用户用可信本地 Python 文件描述连招，入口固定为 `run(player)`。
- 软件必须适合小白：清楚、漂亮、操作少；每次开发先自动检查，再提供简短中文验收。
- 优秀案例 1 只用于核对单键触发、快捷连点、紧凑 UI 和输入安全；不复制其源码、品牌、资源或 JSON/QIM 格式。

## 2. 页面与桌面体验

- 五个页面：宏库、触发、功能、开发连招、设置。
- 宏库：新建、编辑、静态校验、保存、改名、删除、AI 提示词；错误宏不执行，用户注释保留。
- 触发：每个宏可直接修改启用状态、单一触发键、按下/切换、次数和速度，保存后立即生效；状态颜色不能被悬停覆盖。
- 功能：快捷连点必须保持一排；共享游戏键位位于快捷连点下方，名称和物理键都可修改。
- 设置：自定义全局启停键、OSD 开关、提示音开关；不放共享游戏键位。
- OSD 必须按完整提示文字和脚本名称自动扩宽；超过可用屏幕宽度时换行增高，不得从左右裁掉名称，并且不能抢游戏焦点或拦截鼠标。宏启动提示必须显示本次计划循环次数：`COUNT=1`显示“循环1次”，有限多次显示实际数字，`COUNT=0`显示醒目的`+∞`；停止提示只显示宏名和已停止，不得把计划次数冒充已完成次数。
- 窗口、任务栏、EXE 和托盘统一使用用户角色图标；关闭、最小化到任务栏、隐藏到托盘是三个不同动作。
- 程序为无黑色命令窗口的文件夹式便携 EXE；`config/`、`macros/`、`logs/` 位于 EXE 同级并可写。

## 3. 输入、触发与安全

- 每次启动后全局自动化默认启用，设置页立即显示“热键已启用”；启动本身不运行宏，只有真实按下已启用宏的触发键才执行。第一次按当前全局键切换为禁用，再按一次恢复启用。
- 全局键、宏触发键和快捷连点键均支持一个标准键盘键或五鼠标键；不支持两个键组合触发一个宏。
- `down`：物理按下启动，物理松开立即停止；`switch`：第一次物理按下启动，第二次物理按下停止，松开无动作。
- 同一触发键可以绑定多个宏并发运行，每个宏独立停止。
- 自己发送的输入使用 `MAPL` 标记并由监听层过滤，不能再次触发宏、快捷连点、全局开关或提示音。
- 正式 Windows EXE 请求管理员权限，与可能以管理员运行的游戏保持相同权限层级；`SendInput` 返回失败时必须限频写入诊断日志。
- 关闭程序时停止全部播放器和快捷连点、释放输入、注销 hook、关闭 OSD、等待线程，最终不得残留 `MyAutoPlayer.exe`。
- 只允许一个程序实例；第二次启动不得只弹“已经运行”后退出，而必须通过同一Windows会话内的命名唤醒事件，让旧实例恢复、显示并再次尝试前置。

## 4. Python 宏契约

每个宏必须包含：

```python
NAME = "宏名称"
HOTKEY = "mouse_back"
MODE = "down"
COUNT = 0
SPEED = 1.0
ENABLED = True

def run(player):
    player.战技()
```

- `HOTKEY`：一个支持的物理键；不能与当前全局键冲突。
- `MODE`：`down` 或 `switch`。
- `COUNT`：0–99；0 表示由程序重复运行到收到停止请求。
- `SPEED`：0.01–8.0，只缩放 `player.sleep()`。
- `ENABLED`：布尔值。

真实公开 API：

- `player.tap(key, hold_ms=20)`
- `player.sleep(ms)`
- `player.切换(1|2|3)`
- `player.战技()`、`player.声骸()`、`player.大招()`、`player.跳跃()`、`player.处决()`
- `player.按键("当前自定义动作名称", hold_ms=20)`
- `player.mouse_click(button, hold_ms=10)`
- `player.mouse_down(button)`、`player.mouse_up(button)`
- `player.mouse_repeat(count, button, interval_ms=10)`

用户确认的语义：角色 1 默认物理键为 `1`；固定中文函数按槽位工作。动作改名后，可用 `player.按键("改后的名称")`。正确函数名是 `player.战技()`，不是“战绩”。

## 5. 共享动作和连招作者规则

- 固定槽位：角色 1/2/3、战技、声骸、大招、跳跃、处决、扩展键 1/2/3。
- 所有显示名称和物理键可改；显示名称不能重复，物理键允许重复。
- 程序不识别当前角色、能量、资源、动画、技能冷却或游戏状态；连招必须使用用户提供的确定动作和毫秒时间线。
- 项目现有保守作者规则：普通切人前 50ms、快速回切 1080ms、大招后切人 1500ms；用户实测值优先。
- 当前新增角色宏使用：1 号秧秧、2 号琳奈、3 号千咲。旧 JSON 中“卡提”“夏空”“千笑”等名称不采用。

## 6. AI 提示词

- 程序内提示词必须让陌生 AI 只读后即可生成完整、可保存、可中断的 Python 宏。
- 内容覆盖全部真实 API、宏元数据、完整键值、当前全局键、当前共享动作、三角色资料模板、战斗逻辑、示例和安全禁区。
- 当前全局键和共享动作必须在打开窗口时动态读取，不能复制过期默认值。
- 角色事实缺失时外部 AI 只能询问最少必要问题，禁止编造技能、资源、冷却和动画。
- 提示词当前/默认文件改用 UTF-8 Markdown；阅读窗口渲染 Markdown，排版清楚。
- AI 提示词窗口必须非模态：可拖到旁边阅读，同时继续点击宏库、触发、功能和设置。
- `config/ai_prompt.md`是用户可编辑当前版，`config/ai_prompt.default.md`是“恢复默认”备份，两者随正式程序交付。提示词必须解释开发连招会话中的`events.jsonl/events.csv/raw.mp4/metadata.json/native.log`位置、作用和证据优先级，并明确实时“最近事件”仅在内存中，只有开始录像才写入事件文件。
- 提示词必须教AI区分按下/松开、`held_ms`按下时长、`release_gap_ms`松开间隔、`delta_from_previous_event_ms`距上一事件、`overlap_keys`同时按住及`active_keys_after_event`仍在按住；生成宏时六个元数据均有简短行尾注释，`run`按逻辑阶段写中等详细度注释，不逐行复述。
- 可编辑模板自身必须包含第9节完整键盘/鼠标内部值字典，不得只靠程序窗口临时拼接；程序每次打开提示词时把动态第10节当前全局键/共享动作和第11节当前宏源码合成为`config/ai_prompt.complete.md`。界面必须分别显示“完整提示词”和“可编辑模板”的绝对路径，禁止再把基础模板路径误称为程序内完整内容。
- 连续出现且动作、按住时长和间隔完全一致的小片段可用明确次数的有限`for ... range(...)`压缩；不同毫秒节奏必须逐项保留。永久循环仍只能由`COUNT=0`控制，禁止生成`while True`。
- AI提示词窗口必须明确标记：`ai_prompt.complete.md`为每次打开时覆盖的实时只读生成文件，`ai_prompt.md`为持久保存且允许用户修改的模板，`ai_prompt.default.md`为持久默认备份。提供“打开可编辑文件夹”和“打开备份文件存储文件夹”两个按钮；即使当前两者位于同一`config`目录，也要按各自文件路径打开，避免用户手工寻找。

## 7. 视觉与性能

- 当前唯一外观为`樱空花园`主题与`画册侧边导航`布局；经典粉红、经典顶部标签、外观切换器和“恢复经典外观”按用户最新要求移除。
- 樱空花园从用户参考图提取樱花粉、天空蓝、薄荷绿、淡桃橙、云白和柔光卡片感；人物大图不铺在操作区，优先保证表格与控件可读性。
- 画册侧栏顶部使用用户提供素材裁出的圆形动漫头像；侧栏保持紧凑，头像和“宏库/触发/功能/开发连招/设置”全部水平居中，不得挤占主要操作区。
- 宏库“序号”表头与数字必须居中；表格名称列随窗口宽度自动扩展。
- 无边框主窗口支持上、下、左、右和四个角拖动缩放；画册布局最小宽度 760px，启动自动适配为约 840px 且不得超出当前屏幕。
- 首次启动或第二次启动唤醒旧实例时，主窗口必须从最小化/托盘状态恢复并立即可见：先调用Qt显示/激活，再使用`ShowWindow(SW_RESTORE)`、短暂`HWND_TOPMOST`和`SetForegroundWindow`重试；约1秒后取消置顶。Windows可能依法拒绝键盘焦点，但不得只闪任务栏而继续被其它普通窗口完全遮住。
- 触发页和宏库的选中行使用淡蓝背景，但不得覆盖单元格原有文字色；启用保持绿，禁用/错误保持红，普通或紫色文字保持原色。
- 用户角色图标必须尽量填满图标画布，减少透明空白，但不能裁掉头发、发卡或脸部。
- 视觉源文件迁入 `my-automation-tool/assets/`，生成脚本从该稳定位置创建多尺寸 ICO；根临时图标包完成迁移后删除。
- 保持文件夹式 PyInstaller 构建，不退回启动更慢的单文件解包方案。
- 优化目标是减少可控的同步初始化、重复扫描和阻塞等待；不得为追求数字牺牲 hook ready、安全禁用或退出清理。

## 8. 已知问题与证据边界

- 中文输入法下字母输出会进入 IME 组合态；用户已接受英文输入法正确、中文偶发轻微问题，不再阻塞开发。
- 若未来重查中文侧键边沿，只能先记录真实 hook -> FIFO -> 状态机 -> stop 原始证据，再决定是否替换原生 hook；禁止猜改。
- OK-WW 是用户独立程序，不属于本项目残留进程。
- 游戏内无效果要先区分“触发链”和“发送链”：日志已记录宏启动则不得猜改 hook；管理员版仍无效时检查 `SendInput` 返回、前台窗口和游戏过滤。未取得证据前禁止加入或安装驱动输入。
- 外部 QuickInput JSON 只允许一次性、只读转换为本项目 Python 宏；点击、等待和固定循环可转换，图像/OCR、坐标和窗口条件必须明确标为不可等价转换，禁止偷偷改成无条件输入。

## 9. 开发连招原生回放

- 严禁调用、控制或依赖 OBS 及其他成品录屏程序。录像由 MyAutoPlayer 仓库内公开源码的原生模块实现并随正式程序交付。
- 原生路径为用户选择的完整显示器画面（WGC，必要时 DXGI Desktop Duplication 后备）→ Direct3D 纹理 → H.264 编码；不是只录游戏窗口。声音使用 WASAPI loopback。实现可参考 OBS 官方源码的回放缓存、编码线程和拥塞处理架构，但不得复制 GPL 源码或在运行时调用 OBS。
- 编码模式必须由用户明确选择：`GPU（推荐）` 使用可用的硬件 H.264 编码器，`CPU（兼容）` 使用软件 H.264；默认 GPU。GPU 不可用时必须明确报错，不得静默降级到 CPU。界面和会话元数据必须显示实际编码器名称、GPU/CPU 模式、帧率、码率、丢帧和编码拥塞。
- “开始”只启动磁盘滚动分段回放和物理按键旁路记录，不生成正式录像；“保留过往”是唯一允许固化用户所选 5/10/15/30 分钟原始视频及同一时间窗按键日志的入口。保存后继续缓冲，不漏掉后续操作。
- 只有点击“保留过往”才弹出会话文件夹命名框；若输入 `xxx`，最终目录名为 `xxx-YYYYMMDD-HHMMSS-recording`，文件夹内继续使用 `raw.mp4`、`events.csv` 等稳定文件名。禁止只让用户给 MP4 改名；空名称、Windows 非法字符和重名必须给出清晰提示。
- 运行中的主按钮必须明确显示“停止并清空”。点击后不得弹命名框、不得调用导出、不得生成正式会话；原生核心安全结束当前工作后立即删除本次临时滚动分段和临时按键日志，已由“保留过往”固化的历史不受影响。关闭程序同样停止并清理当前临时缓冲。
- 战斗时只编码一份原始视频；保存后只生成外挂ASS/SRT字幕和CSV/JSON。用户明确取消烧录后的`perfect.mp4`，程序不得再计划、生成或把外挂字幕称为“完美视频”。
- 滚动缓存只保存压缩后的短分段，不保存未压缩帧；达到时长后自动淘汰旧段。缓存目录与最终保存目录分离，异常退出时清理临时段但不删除已固化历史。
- 当前 Media Foundation 封装采用约10秒的独立H.264 MP4短段：下一编码器提前准备、旧编码器在单一后台工作器完成封装；捕获回调不得执行阻塞的`new/finish`。导出使用MIT许可`mp4` crate做样本级无重编码重封装，必须校验H.264分辨率、timescale、SPS/PPS、首关键帧和样本时间戳。实际保存起点允许为了可解码关键帧向前扩展至多一个短段，`metadata.json`必须同时记录请求时长与实际时长。
- 用户点击“开始”后，允许的键盘键与鼠标左/右/中/侧键 down/up 在 Windows 全局持续记录，即使用户暂时切到桌面、记事本或 MyAutoPlayer；每条事件同时保存当前前台窗口/PID/进程证据，便于区分游戏与切窗操作。永不记录鼠标移动、文字内容或本程序 `MAPL` 模拟输入。
- 每条事件保存序号、绝对时间、单调时间、相对毫秒、距上一全局事件毫秒、down/up、按下持续毫秒、松开间隔毫秒、事件后仍按住的键、重叠按键、物理键、中文动作名快照和当前前台进程证据。共享键位同键对应多个名称时全部显示，不任意猜一个；这些字段必须能无歧义表达`1 down → Q down → 1 up → Q up`的重叠与细微先后顺序。
- 保存结果按日期/会话分组，至少包含一份原始视频、事件 JSONL、易读 CSV、ASS/SRT 外挂字幕和元数据；历史页可播放、打开目录和删除用户明确选择的会话。
- 历史录像只扫描当前视频保存地址中的日期/会话目录，最新会话排在最上方；正在保存的临时目录不得显示。元数据缺失或损坏时仍保留该行并明确标为“信息不完整”，不能拖垮整个列表。历史页提供刷新、播放`raw.mp4`、打开ASS/SRT外挂字幕、打开会话目录和删除所选会话；删除必须有明确选中项、中文二次确认、再次验证目录未越界，并移入Windows回收站，禁止静默永久删除。
- 实时按键窗消费独立观察队列：首次默认勾选共享常用键、1/2/3、E/Q/R、空格、鼠标左右与侧键，但每一个默认键都允许用户取消；设置窗口列出完整支持键并保留全部勾选状态。非默认键稳定追加在鼠标区域右侧，不得打乱仍选中的默认键布局。
- 实时按键窗采用 OBS Input Overlay 仅作视觉参考的原创布局：背景高度透明、按键为白色轮廓，未按下内部透明；新 down 按樱空花园五色色序显示先后，松开恢复白框。必须去重 Windows 自动重复 keydown，避免重置按住计时或推进色序。
- 最近事件可选显示3、5或10条并持久化；“最近N个事件”、每条详细事件和左下角按住/松开时长属于同一个详情板块，使用统一背景色并通过卡片边框区分事件。窗口顶部显示白字白框透明本地时钟，格式为 `HH:mm:ss:fff`；事件CSV/JSONL也保存同格式本地时间。
- 按键窗口事件卡片使用精简文字：`本地时间 + 按下/松开 + 物理键`。按下卡显示上一次松开间隔、距上一事件及同时按住键；松开卡显示本次按下持续、距上一事件及仍在按住键。左下角“按下/松开”状态计时必须各自真实递增，禁止把按下时长错误填入松开时长。
- 事件卡片必须放入透明纵向滚动区，横向滚动永久关闭。显示时不抢游戏焦点；用户主动点击事件区后可用鼠标滚轮或键盘上下键按固定单步浏览。3/5/10条均适配；窗口缩到50%最小时，长详情自动换成2–3行，不得横向裁切或覆盖其它卡片。
- 最近事件必须按时间线自上而下排列：保留范围内最旧的记录在上，最新记录在最下；超过3/5/10条时只保留对应范围内最新的N条，但不得把它们倒序显示。
- 按键记录窗口的详情板块底部，在“按住/松开”状态右侧提供一个矩形单行临时输入框，灰色占位文字为“可以在此输入按键”。它允许大量文字并以单行横向滚动显示，只作为当前窗口的一次性草稿区；不得写入设置、按键日志、录像或会话元数据，用户点击窗口叉号关闭时必须立即清空。
- 后续待用户完成上述文本框验收后，再单独研究“点击外部视频播放器后，键盘仍继续输入临时框且不影响播放器快捷键”的焦点保持/受控输入方案。本轮明确只记录需求，不修改全局监听、焦点或按键拦截；实现前必须保留鼠标播放操作、关闭即释放和紧急退出边界。
- 详情板块提供5种背景预设：淡黑色、淡白色、淡樱粉、天空蓝、薄荷绿；深浅滑块为0–100%。整体按键窗口的黑色薄背景另有0–100%透明度滑块，不能连同文字和按键一起变透明。所有外观设置持久化，浅色背景必须自动使用足够深的事件文字以保持对比。
- “设置显示按键”必须在打开时同步显示按键记录窗口。按键勾选、最近事件3/5/10条、详情背景、板块深浅和整体透明度均实时预览；右下角提供“确定 / 取消 / 应用”中文按钮。“应用”仅在存在未提交更改时可用，应用后继续留在窗口；取消只回退尚未应用的预览。
- 窗口按内容自适应；新增键多时可缩小按键但不得裁切。四个角分别提供一个精致的8–10px缩放图标，仅鼠标进入窗口时显示，只有按住图标实际区域才允许缩放；字体、间距、按键和圆角等比例变化，最小为推荐尺寸的50%。窗口其它边缘或底部信息区不得误触缩放。关闭时必须记住窗口最后位置和大小，再次打开恢复到原位置；显示器拔除或分辨率变化后，恢复位置必须自动限制在当前可见屏幕内。
- 清晰度只允许720p、1080p，禁止2K。帧率允许15/30/45/60fps，默认30fps；当前性能验收只强制测试15fps，不要求测试30分钟连续录制，但产品仍提供5/10/15/30分钟回放时长。分辨率、帧率、编码器、码率和丢帧必须写入会话元数据。
- 显示器录制必须使用源显示器的物理像素尺寸，再由 Media Foundation 缩放完整画面到720p/1080p；Windows 125%、133%、150%等显示缩放不得造成只录左上角或丢失右侧/底部。元数据同时记录源尺寸与输出尺寸。
- 声音分为系统声音与可选麦克风。开发连招页必须提供清晰的“录制麦克风声音”开关和“选择麦克风设备”按钮并持久化，麦克风默认关闭；设备列表必须来自 Windows 当前真实输入设备，保存稳定设备标识和易读名称。用户可选择设备、开启/关闭、查看权限或设备错误，并用0–200%滑块调整麦克风音量。声音与视频使用同一单调时钟；麦克风标记语音必须进入原始视频并在元数据记录设备、增益、采样率和声道。最终容器必须把桌面/游戏声音保存为音轨1、麦克风保存为独立音轨2，禁止预先混成一条音轨，使剪辑软件可单独静音人声。实现使用项目内WASAPI和多音轨mux，不调用OBS；可借鉴OBS的`audio_encoders[index]`/mixer-mask架构但不得复制GPL源码。在真实音轨接入前，开启麦克风必须阻止录像并明确说明，禁止生成无声视频却声称录音成功。
- WASAPI 的 `data_discontinuity` 只是设备可能发生瞬时缺包的诊断标记，不得单独作为致命错误。时间戳仍可信且缺口不超过2秒时，按同一QPC时间线补静音并继续，分别累计桌面声/麦克风恢复次数写入原生日志、实时状态和会话元数据；时间戳无效、连续缺音超过2秒或时间戳倒退超过2秒时才停止并明确报错，禁止用假音频冒充成功。
- “开始 / 保留过往”右侧提供录制状态按钮：未录制时静止显示，点击开始后以明显但不过度刺眼的樱空花园色调持续闪烁；后台保存时只要缓冲仍运行就继续闪烁，停止或核心失败后立即停止。按钮只作状态指示，不得误触停止录像。
- 开发连招页同时提供“桌面声音（音轨1）”与“麦克风（音轨2）”两个实时峰值状态条。即使尚未开始录像，也必须以轻量WASAPI设备预检持续读取两路真实PCM峰值：播放音乐时桌面条跳动、用户说话时所选麦克风条跳动；预检不得捕获屏幕、保留PCM、编码音频或生成录像文件，并须明确显示“不保存”。“录制麦克风声音”开关只决定正式录像是否写入音轨2，不得关闭空闲麦克风预检。开始正式录像前先停止预检并切换到实际编码电平，停止/失败后自动恢复预检；不得让两套采集同时占用设备。UI轮询不得阻塞音频线程，也不得用模拟动画冒充真实声音。
- Stage 19E字幕从同一原始边沿日志生成：松开事件形成“按下 大写A 50.0ms”，下一次按下形成“松开 大写A 69.0ms”；鼠标左右/中/侧键使用同一规则。字幕时间轴必须保留重叠键并按真实先后显示，不得把同一时刻附近的不同键强行串行化。原始JSONL/CSV继续保留全部高精度字段供AI生成`player.tap/press/release/sleep`脚本。
- Stage 19E字幕卡片固定为两行：第一行明确显示“按键：大写 E/鼠标左键/侧键 1”，第二行显示“按下 50.0 ms · 分开 69.0 ms”。保存会话后生成与`raw.mp4`同名的UTF-8样式字幕`raw.ass`，使用独立可辨认的半透明樱空梅色矢量面板、6px深色描边、2px阴影和最新键高亮，不得只依赖播放器可能忽略的`BorderStyle=3`文字背景；三组信息采用顶部锚定，在1920×1080字幕坐标中与画面底边保留200px安全距离，整体向屏幕中心上移，禁止裁掉最后一组毫秒行。另生成不自动抢占样式的兼容`input_subtitles.srt`。快速连招不得为每个事件创建长期重叠字幕，而应在同一滚动面板保留最近3项、每次只显示一个不重叠时间片；同一10ms ASS时基内的边沿合并为一个最新快照，文本仍保留真实毫秒值。同键下一次down确定上一次分开间隔，录像结束仍未再次按下时明确标注“至录像结束”，未up时明确标注“录像结束仍按住”，禁止伪造未知时长。外挂字幕即最终字幕交付，不再烧录第二份视频。
- 所有面向用户的 Qt 标准弹窗按钮必须使用简体中文，包括录像命名和完成提示、麦克风选择、日志清理、文件选择、确认删除及错误提示中的“确定 / 取消 / 是 / 否 / 保存 / 应用”。不得因系统或 Qt 默认语言显示英文 `OK / Cancel / Yes / No`；代码内部技术标识不受此条限制。
- 默认视频目录为项目可写数据根下的 `captures/`。开发连招页始终显示完整保存地址，同时提供“选择保存目录”和“打开视频保存目录”；用户选择必须持久化。
- 正式版必须自动检测与主程序同目录 `native-replay/myautoplayer-native-replay.exe`；同时显示完整核心路径，并提供“选择录像核心”和“自动检测”。用户可选择另一份合法同名核心，选择需持久化；普通用户不运行 `.ps1`。
- 录像核心必须与 PySide 主界面和宏触发状态机隔离。物理 hook 只在 `MAPL` 过滤后向独立有界非阻塞队列复制轻量事件，不做 Qt、磁盘或编码工作；队列溢出只记诊断，不能阻塞游戏输入。性能验收以游戏帧时间和按键时间线为准，目标是录制增量不破坏用户约 10–50ms 的连招精度。
- 设置页 OSD 必须支持总开关、10–72px 文字大小和淡色半透明背景开关；设置立即生效并持久化。OSD 继续完整显示长脚本名、置顶、鼠标穿透且不抢焦点，背景透明度不得明显遮挡游戏。
- 按键日志保留项目规范键值，同时提供小白可读的完整事件文本：例如“按下了大写 E”“按下了侧键 1”；若该键对应共享动作，则显示“按下了战技（物理键：大写 E）”。鼠标左右/中/侧键、down/up、按住与松开时长均使用同一命名规则。

## 10. 永久不做

鼠标坐标/轨迹、滚轮自动化、游戏内存读取、向游戏注入 DLL/驱动、调用 OBS/成品录屏软件、窗口输入屏蔽、定时无人操作、OCR/图像/颜色识别、角色状态判断、手柄、变量流程、案例 JSON/QIM、两个键组合触发、联网技能数据库和自动更新。项目内原生屏幕/声音回放与用户本人允许键的旁路日志是唯一获准的录制范围。
