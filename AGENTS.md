# MyAutoPlayer：50% 减负模式

## 最高指令：按需案例专家

`案例专家`是永久保留的唯一子 Agent。负责人遇到无法由当前源码、规格和自动测试解决的技术问题、案例行为或代码设计难题时，必须调用他；他可只读查看“优秀案例 1”“优秀案例 2”的全部源码，并结合本项目全部逻辑给出源码证据、适配建议和未知项。没有这类难题时，负责人自行完成工作，不调用任何子 Agent。案例专家不复制案例源码/资源、不直接改代码、不扩大产品范围。

默认只读本文件、`my-automation-tool/docs/handover/CURRENT_HANDOVER.md`，并运行 `git status --short`。只有开始新功能时才读取该功能的一页规格和验收教程；旧员工文件、编号交接、审查日志和旧路线已从工作区删除，需要追溯时只从 Git 标签读取。

负责人单上下文完成：明确一个小目标和范围外 → 简短 READY 判断 → 实现 → 自动测试/编译/差异检查 → 中文 Windows 教程 → 用户验收 → 更新当前交接。无需默认调度员工、重复审查或复制历史文字。

保留的底线：Python `run(player)`、F9/F12、F2 占位、停止释放、真实输入先自动测试后人工验收、案例只读、用户文件不动、默认不发布且禁止强推。只有无法确认的案例/UI/安全事实才按需请一名专家。

这比上一版默认四份文档加多员工流程减少超过 50% 的阅读和等待；平台实际模型以运行环境为准。

## 最高指令：优秀案例 1 源码先行与零偏移

优秀案例 1 的完整源码已确认位于 `my-automation-tool/优秀案例1-Quickinput/Quickinput-main/source/`，包含 `src/trigger.cpp`、`src/macro.cpp`、`src/thread.cpp`、`ui/TriggerUi.cpp` 和 `ui/widget/QKeyEdit.h`。后续 AI 不得因目录预判、工程文件引用或一次失败的检索而声称源码缺失；开始设计、排障、验收规划或写代码前，必须先对照这个真实目录和对应实现。优秀案例 2 只在案例 1 无对应证据时补充查阅。

用户只要键盘连招、鼠标点击/长按/侧键和已明确要求的 UI。每一项实现须先找到对应案例行为和命名，再适配到可信 Python `run(player)` 架构；不复制 C++ 源码、资源、品牌、JSON/QIM 格式，也不加入用户未要的案例功能。无法找到时先由案例专家用准确路径检索；仍无法确认才阅读本文件与当前交接，并明确记录 `UNKNOWN`，禁止猜测实现。

发生“未核实源码即设计、错误声称源码不存在、实现与用户已确认案例行为不一致”即为偏移。负责人必须立刻停止扩展、纠正实现、在 `my-automation-tool/docs/handover/ALIGNMENT_RECORD.md` 记录责任、扣分和纠正证据；后续负责人先阅读该记录引以为戒。该记录是流程质量控制，不影响用户的 Git 历史或已验收版本。

## 最高指令：小白验收文档质量

每个需要 Windows 人工验收的阶段，必须提供一份**漂亮、简短、清晰、面向小白**的中文 Markdown 验收文档。至少包括：醒目的安全准备、按顺序编号的操作、每一步可观察的正确结果、明确通过标准、简短提示、可直接复制的失败反馈模板和未覆盖范围。优先使用短句、表格、留白和有限的视觉提示帮助阅读；不得堆砌内部术语、测试原理、长段落或要求用户猜测下一步。发布前负责人必须自行检查路径、启动方式、按钮名称和预期文字是否与当前实现一致。
