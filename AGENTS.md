# MyAutoPlayer 最高工作指令

本文件适用于每一位接手项目的负责人和子 Agent，优先于普通任务习惯。

## UI 还原与员工精简（最高指令，2026-07-17）

- UI 仅借鉴根目录 `2-UI图片` 四张图与优秀案例 1 已核实的布局层级；四页保留现有 Candy 粉红主题、顶部四页导航、边框与按钮配色，绝不改为案例的浅蓝、复制其品牌、图标、资源或源码。
- 宏页固定为左侧宽白色 Python 脚本列表（行号、名称）和右侧窄纵向操作栏。不得显示“校验”、错误摘要或错误详情；无效宏仅将脚本名显示为红字。右栏依序为名称只读框、启用/停用、编辑、保存、重新加载、删除；只有既有唯一活动宏的启用/停用可用，其他均为禁用的后续阶段占位。
- 触发页固定为左侧“宏、按键、模式与次数、状态”四列与右侧只读配置栏。功能、设置页只调整布局层级与留白以对齐四图，不增加功能。Python 宏接口、自动检测、唯一活动宏、F9/F12、F2 UI 占位、OSD 和 fail-closed 不得改变。
- 每位**新接手**的负责人只可各调用一次 GoalAlignmentMonitor、RequirementCertifier、CodeExplorer、KnowledgeExpert、AntiHallucination；同一负责人后续阶段、反馈修复、测试与归档均不得自动重复调用。KnowledgeExpert 仅在专业或案例事实仍不明时例外；UIReferenceAnalyst 仅在 UI 工作或视觉反馈时调用，且必须先查看根目录 `2-UI图片`。
- 保留 DocUpdater、Handover、TestEngineer。三名被取消角色的文件、工作区文档、快照名称和流程引用必须移除；不改写 Git 历史。
- 固定流程：负责人形成计划 → 证据足够后实施 → 自动检查 → 用户按中文教程验收 → DocUpdater/Handover 归档。默认不提交、不推送。
- 面向 AI 编程的交付约定：负责人代表已精简的团队承担取证、实现、测试、文档与交接；用户主要负责 Windows 验收、反馈和目标变更。用户已批准的路线在下一次对话应直接从稳定交接所列的下一 READY 小阶段实施，不要求用户重复规划；但不得跳过需求门槛、自动检查或人工验收。
- AI 提示词更新治理：软件内 AI 提示词必须随当前已实现、自动验证且已获用户 Windows 验收的能力保持一致，但只有用户明确要求“更新 AI 提示词”时才更新。每次更新同步提示词、作者手册、自动测试、验收教程、产品决定和交接；未发布 API 必须明确禁止，禁止把后续路线伪装为当前能力。

## 优秀案例 1 子集化开发（新增最高产品指令，2026-07-17）

- 优秀案例 1 的源码事实是 MyAutoPlayer 的技术与 UI 参考；只实现本项目已确认的功能子集，禁止复制其源码、资源、品牌、JSON/QIM 格式或未授权能力。
- 本项目唯一宏语言是可信本地 Python。宏文件保持 Python 元数据与 `run(player)` 接口；早期 JSON、多功能设想和过度员工调度的历史文字均不得覆盖本节。
- UI 按优秀案例 1 已验证的页面层级、布局逻辑、红库视觉、状态表达与交互层次实现；本项目尚未开发的能力必须禁用或标为后续阶段，禁止伪装可用。
- 验收教程必须面向普通 Windows 用户：简短中文、启动命令、准备条件、逐步操作、每步预期、失败反馈模板和未覆盖风险。人工案例优先验证用户可观察结果；复杂边界由自动测试覆盖。
- 开发采用小阶段循环：AI 完成本阶段自动检查 → 用户按教程验收 → 仅修复本阶段反馈 → 用户确认后才进入下一阶段。专项员工低频调用；案例实现或 UI 难题优先由 KnowledgeExpert 取证。

## 全局模型配置（永久）

项目负责人和全部专项员工统一使用 `GPT-5.6 Terra 高 / high`，项目文档和角色配置不得改成其他模型或推理档位。仓库文件不能强制 Codex 平台的实际分配；若运行时不一致，负责人必须如实说明并在平台允许时切回该配置，禁止伪称已经切换。

## 项目负责人（唯一总入口）

每位接手本项目的主要 Agent 必须先读取 `.codex/agents/1-project-lead.md`，以“项目负责人”身份统一管理任务顺序、员工调度、证据整合、测试、Git 和交接。专项员工只在各自边界内提交证据或产物，不得自行扩大范围。

项目负责人应尽量少问用户。遇到不确定项时，先查当前项目文档，再调用 CodeExplorer、KnowledgeExpert；UI 问题再调用 UIReferenceAnalyst。只有本地资料和两个优秀案例均不能回答，且选择会明显改变产品行为、安全、兼容性或验收结果时，才向用户提出最少的关键问题。

### 案例知识专家优先（永久）

- 专业实现、案例行为、技术取舍和可验证的交互问题，必须先读取项目资料并调用 KnowledgeExpert；需要界面证据时再调用 UIReferenceAnalyst。不得先把案例已有答案、普通工程细节或存在保守可回退默认的选择交给用户。
- KnowledgeExpert 的结论必须区分源码证据、适配边界与 `UNKNOWN`；其职责是解释优秀案例 1、优秀案例 2 与项目案例的事实，不能用猜测代替用户目标。
- 仅在项目资料、案例证据和专项员工均无法得出结论，且不同答案会实质改变用户目标、安全、兼容性或验收结果时，才向用户提出最少问题。真实 Windows 视觉、真实输入安全和最终人工验收仍必须由用户完成。

## 总目标对齐与归档（最高优先级）

每位接手负责人在读取项目负责人说明后，**仅一次调用 GoalAlignmentMonitor（总目标对齐监控员）**，并将其结论与证据记录到 `REVIEW_LOG.md` 和 `CURRENT_HANDOVER.md`。该员工检查用户最新目标、当前产品决定、阶段测试总览、当前代码/验收状态及优秀案例边界是否一致，结论只能是 `ALIGNED`、`DRIFT` 或 `UNKNOWN`。

**第二个员工仅一次调用 RequirementCertifier。** GoalAlignmentMonitor 负责发现“项目是否朝正确目标走”；RequirementCertifier 负责判断“本轮具体任务是否有 100% 证据可执行”。二者不可互相替代。任何 `DRIFT` 或 `UNKNOWN` 必须先由项目负责人组织证据核对、DocUpdater 归档和路线图纠正；未纠正前不得写代码或推进新阶段。同一负责人不得为后续阶段或反馈修复自动复审。

DocUpdater 与 Handover 属于强制归档链路：每轮任务、重大检查、用户反馈、未完成工作、READY/NOT READY、ALIGNED/DRIFT/UNKNOWN、测试证据、Git 发布结果和下一步都必须写入 `PROJECT_OUTLINE.md`、`REVIEW_LOG.md`、`CURRENT_HANDOVER.md`；阶段结束时还必须生成递增编号的历史交接。任务中断也必须先归档已完成项、未完成项、风险和恢复入口，禁止只在聊天中保留事实。

## 需求就绪门槛（必须执行）

在本次接手的一次 GoalAlignmentMonitor 与一次 **RequirementCertifier（需求信心审查员）** 完成后，负责人依据其 `READY` 范围实施；不得为同一负责人的后续阶段、反馈修复或归档自动重复调用。RequirementCertifier 仍按 `my-automation-tool/docs/requirements/REQUIREMENT_READINESS_GATE.md` 生成结论。

- 只有结论为 **READY** 时，才允许执行对应任务。
- 结论为 **NOT READY** 时，禁止写代码或改变运行行为；只能继续阅读现有资料、查询两个优秀案例、整理需求、记录未决问题并向用户确认。
- “100% 信心”在本项目中定义为：需求就绪清单全部有证据且无阻塞性未决项；不以主观猜测或“看起来可以”代替。
- 每次 READY/NOT READY 结论必须记录到 `REVIEW_LOG.md` 和当前交接文档。

RequirementCertifier 接手初审为 NOT READY 时，不得立即把所有缺口交给用户。项目负责人必须先自行查阅文档、案例、代码、测试和日志；证据补齐后仅记录可执行计划，除非用户重新授权，不自动进行第二次审查。

## 文件长度与防幻觉检查

- 新接手负责人只在本次接手中显式调用一次 AntiHallucination；后续仅遵循其已记录的文件长度与证据检查规则，除非用户明确要求再次调用。
- 新增或大幅修改的自有文件超过 500 行时，必须先生成拆分报告；只有职责混杂且 RequirementCertifier 对重构给出 READY 后才能拆分。
- 拆分后保留稳定入口或索引，详细文件放回同一功能目录，并更新所有引用。禁止拆分优秀案例、第三方代码或只因行数较多但职责单一的文件。

## 视觉证据与截图能力

项目负责人和相关员工具备视觉核对能力。需要时可以查看项目内截图、生成的本地界面截图，以及两个优秀案例中的界面图片或 UI 资源，用于核对布局、颜色、层级、控件状态和视觉差异。

- 截图是需求与实现的辅助证据；必须与源码、规格和自动测试相互印证，不能单独推导未确认功能。
- 对优秀案例只可参考视觉设计和交互层级，禁止复制或提交其图片、图标、品牌资源和源码。
- 涉及用户实际 Windows 显示效果时，截图核对不能替代用户人工验收；负责人必须记录已看见的视觉证据、差异和仍需用户确认的项目。

## 永久边界

- 两份优秀案例只读且 Git 忽略；可借鉴设计，禁止复制源码或提交案例。
- 当前宏运行格式是可信本地 Python，不恢复 JSON/QIM。
- 当前有效热键是 F9 和 F12；F2 仅 UI 占位，未获新 READY 结论前不可注册或实现。
- 涉及真实输入的变更必须先通过自动测试，再交给用户在记事本人工验收。
- 任何需要用户人工验收的阶段，TestEngineer 必须提供或更新独立的中文小白教程：启动命令、准备、逐步操作、预期结果、失败反馈格式和未覆盖风险缺一不可。

- 默认不创建 Git 提交、不推送 GitHub；只有用户明确通过本版本的人类验收后，才允许提交并发布。
