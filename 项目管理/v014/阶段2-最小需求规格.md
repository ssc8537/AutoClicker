# 阶段 2 最小需求规格

> 状态：需求已对齐，尚未实现 | 日期：2026-07-15
>
> 本文档仅规定阶段 2；与旧的长期规划冲突时，以本文档为准。

## 一、要解决的真实问题

用户要在游戏外手动编辑一个宏文件，按设定热键后，把固定的按键序列快速、按顺序发送到当前目标窗口。阶段 2 只用 `Hello World` 验证这条链路，不宣称已经实现游戏连招。

安全底线不变：程序启动时禁用；F12 可停止正在运行的宏；鼠标位于程序窗口内时不触发；关闭窗口时停止并释放热键。

## 二、本阶段唯一交付物

1. 一个可加载、会校验的 JSON `Hello World` 测试宏。
2. 一个后台运行、可立即请求停止的 `SequencePlayer`。
3. 将这一个宏接入现有热键管理器，使用其已有的 `switch`、`down`、`up` 三种模式。

没有主界面或编辑器；用户直接编辑 JSON 文件。没有窗口匹配、音效、第二个宏、热键冲突检测、辅助组合键或多宏并发。

## 三、唯一 JSON 格式

阶段 2 只接受一个 UTF-8 JSON 对象，字段必须全部存在且不得有未知字段：

```json
{
  "name": "Hello World",
  "hotkey": "f9",
  "mode": "down",
  "count": 0,
  "speed": 1.0,
  "enabled": true,
  "steps": [
    {"type": "text", "value": "Hello World", "delay_ms": 50}
  ]
}
```

字段规则：

| 字段 | 规则 |
|---|---|
| `name` | 非空字符串；阶段 2 测试值固定为 `Hello World`。 |
| `hotkey` | 单个 `keyboard` 键名；阶段 2 测试值固定为 `f9`。 |
| `mode` | `switch`、`down` 或 `up`；分别为按下切换、按住运行、松开运行。 |
| `count` | 整数 `0` 至 `9999`；`0` 表示无限循环，其他值表示完整序列轮数。 |
| `speed` | 数字 `0.1` 至 `10.0`；`2.0` 使等待时间减半，`0.5` 使等待时间加倍。 |
| `enabled` | 布尔值；`false` 时不注册该宏。 |
| `steps` | 非空数组；阶段 2 只允许 `text` 步骤，`value` 为非空字符串，`delay_ms` 为 `0` 至 `60000` 的整数。 |

`speed` 的计算是 `实际等待毫秒 = delay_ms / speed`。这纠正了旧知识文档示例中乘以速度的相反写法。

## 四、执行与停止语义

- 每一轮按 `steps` 的顺序执行；步骤后的等待也必须可中断。
- `down`：物理按下只启动一次；松开立即请求停止。若 `count` 为非零，执行完指定轮数后自然结束。
- `switch`：按下在开始与停止间切换；`count=0` 可持续循环。
- `up`：松开后执行指定轮数；若 `count=0`，运行至 F12 或关闭窗口停止。
- F12、窗口关闭和每个模式的停止动作都必须调用同一停止路径。停止后不得开始新步骤；已发出的输入按现有输入模拟器的释放逻辑完成释放。
- 同一时刻只允许这一个测试宏运行；再次启动请求的确切策略为拒绝并记录日志。

## 五、验收边界

自动化测试必须覆盖：JSON 合法与非法校验、顺序执行、速度换算、等待可中断、重复启动被拒绝、三种模式的启动/停止回调语义。界面与真实键盘测试由用户执行。

用户手动验收时，仅需按既有教程启动程序，在记事本中验证 F12、F9 和停止后无后续输出。实际游戏连招、鼠标动作、组合键和 10ms 延迟精度不属于阶段 2 的验收。
