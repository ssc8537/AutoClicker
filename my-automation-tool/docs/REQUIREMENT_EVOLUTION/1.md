# 可行性分析与最终技术路线

## 1. 罗技鼠标宏开源项目（Lua）是否可行？

**结论：不可行，应放弃此路线。**

- 你的鼠标是**普通鼠标**，不具备罗技G系列鼠标的板载Lua虚拟机。
- 罗技宏项目（如GHub Lua脚本）依赖罗技官方驱动和鼠标硬件，无法运行在普通鼠标上。
- 强行寻找star高的罗技宏开源项目并修改，会陷入死胡同：底层依赖罗技SDK，普通鼠标根本不会被识别。

## 2. 你的真实需求重新锚定

你要做的是一个**独立的键盘自动化序列播放器**，而不是鼠标宏。

核心逻辑：
- 按下键盘快捷键 → 顺序发送一系列键盘按键（1,2,3,E,R,Q,空格,鼠标左/右键） → 可配置的延迟和循环次数。
- 支持多脚本同时运行（模拟多角色同时在场）。
- 具有UI配置界面，小白友好。
- 参考`Quickinput`的UI和热键绑定逻辑，参考`ok-ww`的Python输入模拟实现和模块化架构。

因此，正确的技术路线是：
**Python + PyQt5/PySide6 + pynput/keyboard（Windows API模拟输入）独立应用程序**。

## 二、模块职责与交互规则（任何修改必须遵循）
1. **input_simulator.py**：只负责单次按键/鼠标点击，不涉及序列逻辑。公开函数：press_key(key, duration=0.05), release_key(key), tap_key(key, duration=0.05), click_mouse(button='left')。内部使用ctypes调用SendInput保证游戏兼容性；pynput作为备用。
2. **sequence_player.py**：接收标准化序列 list of (action_type, value, delay_ms)，执行时支持外部中断标志（threading.Event）。提供play_sequence(seq, speed_factor=1.0, stop_event)方法。执行期间通过信号与UI通信（状态栏更新）。
3. **script_engine.py**：加载scripts文件夹下所有.py和.json文件。对于.py，要求必须包含def run(player):函数，player对象提供press(key), sleep(ms)等方法；对于.json，根据json结构生成序列。统一返回 (script_name, sequence_list, metadata)。
4. **hotkey_manager.py**：使用pynput或keyboard库注册全局热键。检测鼠标是否在主窗口区域内，若在则抑制触发。提供全局禁用/启用热键。
5. **main_window.py**：组装UI，连接信号槽。不直接处理任何底层模拟，全部通过Core模块调用。脚本执行必须在单独线程中，防止UI冻结。
