# 下一位 AI：产品方向与 Quickinput UI 参考规格

> 状态：v020 规格，尚未实现 UI。依据：本项目保存的图片 1–4 与 Quickinput `source/ui/*.cpp/.ui` 源码。图片 5“关于/更新”明确舍弃。

## 一、产品方向与硬边界

本项目是自己的 Python/PySide6 自动化工具，借鉴 Quickinput 的页面结构、交互层级和热键框架；不是复制 Quickinput 的源码、品牌、JSON/QIM 文件格式或完整功能集。

- 宏库：保留这个产品名称，含义是“分组管理的 Python 脚本资源库”，不是 JSON 编辑器。
- 运行格式：可信本地 `.py` 脚本；当前只有 `scripts/hello_world.py`。
- 当前可运行热键：F9 运行脚本，F12 全局启用/禁用并显示 OSD。
- F2：只预留 UI 位置，不能注册、不能显示新 OSD、不能影响 F9/F12。
- 触发模式：只保留 `switch`、`down`；不可恢复 Quickinput 原有的 `up`。
- 当前明确排除：鼠标执行、快速连点、窗口匹配/激活、录制、音效、定时、OCR、输入驱动、开机自启、图像识别、角色 AI、多脚本并发、JSON/QIM 运行格式、图片 5 页面。

## 三、页面 1：宏库

源码证据：`优秀案例1-Quickinput/Quickinput-main/source/ui/MacroUi.cpp:113-226,279-321,407-428`；编辑器下游：`EditUi.cpp:295-363,376-550,1985-2115,2826-2833`。

Quickinput 的宏库读取 `Qi::macroGroups`，以“分组 → 宏文件”展示资源；创建、录制、重命名、导入导出、重载和删除都围绕宏文件工作。其 JSON/QIM 只是原项目存储格式，不是宏库的定义。

| 原版控件 | Quickinput 实际行为 | 本项目后续决定 |
|---|---|---|
| 分组/宏表格 | 展示分组和每行一个宏，选择状态控制按钮可用性 | 保留；数据源改为 Python 脚本库 |
| 录制 | 录制全局动作并保存宏 | 仅保留未来入口位置；不实现 |
| 录制窗口 | 选择目标窗口后录制 | 不做（依赖窗口匹配） |
| 添加 | 建立空的 `down/count=1` 宏并保存 | 未来创建 Python 脚本模板 |
| 编辑 | 加载专用动作编辑器 | 未来改为外部/简易 Python 脚本编辑入口；不复制动作 DSL 编辑器 |
| 导出/导入 | JSON/QIM 文件导出或复制导入 | 当前不做；未来只讨论 `.py` 或脚本包 |
| 重新加载 | 重扫宏目录，刷新表格 | 未来保留，扫描脚本目录 |
| 添加/删除分组 | 操作目录和分组集合 | 未来保留；删除策略单独设计 |
| 删除宏 | 送入系统回收站 | 未来保留此安全意图，不直接删除 |
| 双击重命名/拖放 | 重命名目录或文件、调整组织 | 后置，不进入 UI 外壳阶段 |

原版 EditUi 是三段动作序列（进入/运行/结束）编辑器，覆盖图像、OCR、窗口、声音和变量等近 3000 行能力；本项目不能移植。Python 宏的 `run(player)` 才是唯一脚本主体。

## 四、页面 2：触发

源码证据：`TriggerUi.cpp:16-36,51-179,181-280`；状态批量开关：`116-141`；保存模型：每次字段更新后调用 `currentMacro->save()`。

| 原版控件 | Quickinput 实际行为 | 本项目后续决定 |
|---|---|---|
| 宏/分组表与状态列 | 选中宏加载表单；可开关单宏或整组 | 未来保留脚本启用状态；不可与 F12 全局状态混同 |
| 触发模式 | `switch/down/up` 三种 | 只显示/接受“切换、按下执行”；拒绝 `up` |
| 触发按键 | 最多两枚键盘/鼠标/手柄键 | UI 外壳期显示 F9 固定值；未来多脚本阶段再设计 |
| 运行次数 | 0 为无限，正数为有限轮数 | 保留语义；当前 Python 为 `COUNT 0–99` |
| 运行速度 | 调整等待和鼠标轨迹速度 | 保留键盘等待语义；当前 Python 为 `SPEED 0.01–8.0` |
| 屏蔽触发键/鼠标 | 运行时吞掉输入 | 不做 |
| 脚本变量 | 编辑原版宏表达式 | 不照搬 Python 为单行表达式；后置 |
| 窗口绑定、定时、缩放、偏移 | 窗口规则、时段与鼠标参数 | 明确不做 |

现有语义必须不回归：`down + COUNT=0` 在松开 F9 时停止；有限次数即使松开也完成轮数；`switch` 为首次启动、再次停止；F12 随时停止。

## 七、实施顺序和验收

1. 先制作四页 UI 外壳、红库主题、标题栏和禁用状态，不改变 `main.py` 的输入运行路径。
2. 自动回归现有 16 项测试，新增仅验证 UI 控件存在/禁用状态的测试。
3. 用户对照四张图片检查外观，再用记事本回归 F12/F9；F2 必须没有反应。
4. 视觉验收后，单独规划 Python 宏库扫描与多脚本热键；不与 UI 外观阶段混做。
