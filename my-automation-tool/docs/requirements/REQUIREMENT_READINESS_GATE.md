# 需求就绪门槛

## 目的

让用户可控地推进项目：在负责人 100% 理解为“所有实施所需决定都已被证据锁定”之前，不允许写代码。该门槛不承诺现实世界绝无 bug；它保证不在需求不清时凭空选择产品行为。

## 强制流程

1. 新负责人接手、用户提出新需求、准备修改运行代码或启动新阶段时，先调用 RequirementCertifier。
2. 审查员读取规格、交接、测试、当前代码和必要案例证据。
3. 输出 READY 或 NOT READY，复制到 `REVIEW_LOG.md` 与 `CURRENT_HANDOVER.md`。
4. READY：负责人仅执行审查明确允许的范围。
5. NOT READY：负责人只补需求资料或向用户提出最少的关键问题；不得写代码、改配置、创建功能 UI 或改变热键。

## READY 清单

| 项目 | 必须具备的证据 |
|---|---|
| 目标 | 用户可理解的一句话价值与成功结果 |
| 范围 | 本阶段做与不做的明确列表 |
| 行为 | 用户操作、界面/热键反应、数据流和保存时机 |
| 安全 | 停止、异常、无效输入和回退行为 |
| 接口 | 受影响文件、公开 API、数据或 UI 控件 |
| 测试 | 自动测试、人工步骤和通过标准 |
| 冲突 | 旧文档、案例或现状的矛盾已消除 |

任一项缺失、存在多个合理实现却未选择、或案例证据不足，即为 NOT READY。
