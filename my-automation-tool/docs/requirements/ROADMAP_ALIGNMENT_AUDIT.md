# 阶段路线图与优秀案例 1 对齐审查

> 日期：2026-07-16
>
> 结论：文档纠偏 **READY**；依据旧阶段表直接实现后续代码 **NOT READY**。

## 审查结果

当前主线没有偏离：本项目是可信本地 Python 宏工具，以 F9/F12 安全、可取消执行和 Quickinput 风格的粉红四页 UI 为核心。偏离出现在旧测试路线图：它把历史 JSON、任意热键、多脚本并发、窗口匹配、音效和鼠标执行写成了默认必经阶段。

因此，“旧表全部通过”不能证明实现了用户心目中的完整项目；它会同时漏掉已验收的 Python 宏运行时，又暗中承诺未确认能力。

## 可从优秀案例 1 借鉴的部分

| 案例证据 | 本项目可借鉴 | 本项目当前边界 |
|---|---|---|
| `trigger.cpp`、`thread.cpp` | 全局状态优先、`switch/down`、可中断后台执行 | 不恢复 `up`，维持 F9/F12 与单实例 |
| `MacroUi.cpp`、`TriggerUi.cpp`、`FuncUi.cpp`、`SettingsUi.cpp` | 宏库/触发/功能/设置的页面层级与禁用态 | 数据源为可信 `.py`，不复制 QIM/JSON 或案例 DSL |
| `MainUi.ui`、`init.cpp` | 642×510、垂直缩放、Candy 色、字体与导航层级 | 不复制案例资源、主题切换或窗口持久化 |

案例中的 JSON/QIM、任意两键/鼠标/手柄热键、`up`、窗口绑定、录制、鼠标、音效、托盘和多宏调度均不能因“案例已实现”而自动成为本项目需求。

## 对“完整项目”的诚实结论

完成当前已锁定主线，可交付一个安全的单 Python 宏工具与粉红四页 UI 外壳；它不等于 Quickinput 的完整复制品。

“最终完整项目”仍需用户明确：是否纳入多 Python 宏、每宏热键、受控编辑/持久化，以及窗口匹配、音效、鼠标、录制、定时等候选能力。在这些决定前，任何阶段表都只能把它们列为候选，不能承诺“全部完成”。
