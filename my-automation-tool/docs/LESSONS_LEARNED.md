# 经验教训：自动化按键项目的进程管理

> 本文档记录了一次真实事故的复盘：删除了运行中的 Python 自动化脚本源代码，导致孤儿进程在后台持续运行十几个小时，鼠标侧键和小键盘 0 键不受控制地触发自动按键序列。这份经验可以被另一个 AI 自动化项目直接引用，避免同类问题。

## 2026-07-17：窗口文本应按读者而非 Markdown 编写

可复制提示词窗口是普通文本控件，不能依赖 Markdown 标记表达层级。验收反馈应作为已通过阶段的最小修复：移除框架中的标题、列表和代码围栏，同时保留用户源码本身的注释，既改善阅读也不破坏完整源码复制。游戏按键名称必须以用户最新决定为准，并在提示词、作者手册和验收教程中同步。

---

## 2026-07-17：AI 辅助应复制完整上下文，但只暴露已发布接口

AI 提示词窗口只展示可复制文本，不执行、导入或静态分析之外地处理宏源码。把已保存完整宏、用户需求和“返回完整文件”的要求放在同一提示词中，能让用户安全地完成“复制—询问 AI—粘回编辑器—保存—验收”闭环；同时必须明确只允许已发布 `player.tap` 与 `player.sleep`，禁止 AI 臆造鼠标、游戏角色语义或程序热键接口。

---

## 2026-07-17：脚本管理与触发权限必须分层

优秀案例 1 的宏页只负责资源选择和文件编辑入口；真实可触发集合由触发页每条宏的状态控制。MyAutoPlayer 适配时，宏页选中脚本不得改变运行时选中路径；触发页必须完整投影已扫描 Python 脚本，只有有效行的状态列能启用/禁用唯一活动宏。这样既保留“资源管理”和“触发控制”的可见边界，也不恢复案例的多宏并发、JSON/QIM 或编辑保存能力。

---

## 2026-07-17：小阶段闭环能避免重复验收和范围膨胀

Stage 2A 先以静态校验、原子保存和自动回归建立安全底座，再由用户按中文教程完成 Windows 文件操作与 F9/F12 验收。用户通过后，稳定交接、测试总表和审查日志必须立即改为“已验收”，不能让下一位 AI 重复索要相同证据。后续编辑器高亮、缩进和中文文案被独立为 Stage 2B，明确不夹带游戏键位或鼠标输入；这样每轮都能给用户一个小而可观察、可反馈的程序版本。

---

## 2026-07-17：Python 宏编辑必须先静态校验，再原子保存

宏库的扫描、编辑和重新加载都不能执行用户宏顶层代码。Stage 2A 将 `NAME/HOTKEY/MODE/COUNT/SPEED` 与严格 `run(player)` 作为 AST 静态门槛；只有临时文件写入并再次静态校验成功，才用原子替换覆盖旧文件。重命名同时更新文件名与 `NAME`，而且活动宏必须先在触发页禁用。这样语法错误、接口错误或写入失败都不会破坏原来的可用宏。

## 一、事故经过

1. 你开发了一个 `GameMacroManager.pyw`（游戏连招宏管理工具），使用 Python + tkinter + Win32 API 实现全局热键轮询，绑定 **XButton1（鼠标前侧键）**、**XButton2（鼠标后侧键）**、**Numpad0（小键盘 0）** 触发自动按键序列。
2. 在程序**仍在后台运行**的情况下，你删除了源代码文件。
3. 数月后复原代码，发现按下上述按键仍然触发自动操作，但找不到地方关闭。
4. 排查发现：4 个 `pythonw.exe` 孤儿进程（来自另一个已删除的 `QiAutoClicker.pyw`）在后台存活了 12+ 小时，持续执行全局按键监听。

---

## 二、根因分析

### 2.1 Python 进程的生命周期特性

- `pythonw.exe`（而不是 `python.exe`）不会显示控制台窗口，用户完全感知不到它在后台运行。
- 即使源代码文件被删除，Python 进程不会因此退出。解释器已将代码加载到内存中，`.pyc` 缓存文件可以继续提供字节码。
- `Get-Process` 看到的进程命令行仍指向原文件路径，即使原文件已不存在。

### 2.2 用户感知断层

- GUI 窗口被关闭 != 进程退出（如果程序使用 `pythonw.exe` 启动且后台线程未退出）。
- 没有系统托盘图标或通知，用户无法直观判断程序是否还在运行。
- `macros.json` 配置文件为空（`[]`），说明用户删除代码前未保存配置，但也反映了配置持久化的薄弱环节。

## 三、诊断与解决方法（Windows PowerShell）

```powershell
# 1. 查看所有 python 相关进程
Get-Process | Where-Object { $_.ProcessName -like '*python*' }

# 2. 查看进程的完整命令行（确定每个进程的来源）
Get-WmiObject Win32_Process | Where-Object { $_.Name -match 'python' } |
    Select-Object ProcessId, Name, CommandLine

# 3. 查看进程启动时间（判断是否异常存活）
Get-Process pythonw | Select-Object Id, StartTime, CPU

# 4. 杀掉所有 pythonw 进程（解决孤儿进程问题）
Get-Process -Name pythonw | Stop-Process -Force

# 5. 验证是否清理干净
Get-Process -Name pythonw -ErrorAction SilentlyContinue
```

> `Get-Process -Name pythonw | Stop-Process -Force` 是最直接、最有效的解法，一条命令解决所有弃用进程。

## 四、给自动化项目开发的架构建议

### 4.1 进程可视化管理

| 机制 | 说明 |
|------|------|
| 系统托盘图标 | 使用 `pystray` 或 Win32 API 在系统托盘常驻图标，右键可关闭/重启/查看状态 |
| 进程信号文件 | 启动时创建一个 `.lock` 或 `.pid` 文件，退出时删除。用户或另一个程序可通过检查该文件判断是否在运行 |
| 任务管理器可见 | 避免使用 `pythonw.exe` 启动关键工具，或用自定义进程名（如 `--process-name`）让用户能识别 |

### 4.2 热键开关设计

GameMacroManager 的设计是正确的方向，但可以做得更好：

- **全局开关必须醒目**：工具栏的 "Global: ON/OFF" 按钮是核心控件，建议使用颜色区分（绿色=开启，红色=关闭）而不是纯文字。
- **热键切换**：提供一个硬编码的紧急停止热键（如 `Ctrl+Alt+F12`），任何时候按下都能强制停止所有热键轮询。
- **窗口关闭 = 完全退出**：`WM_DELETE_WINDOW` 事件处理中，确保先停止所有工作线程，再销毁窗口，最后退出进程。当前代码已经做到了这一点：
  ```python
  def on_close(self):
      Config.save()        # 保存配置
      HotkeyPoller.enabled = False  # 停止热键轮询
      self.root.destroy()  # 销毁窗口
  ```

### 4.3 进程防孤儿策略

- **atexit 注册清理函数**：在 Python 中注册 `atexit.register(cleanup)` 确保进程退出时释放全局钩子。
- **看门狗机制**：主线程监控子线程心跳，如果 GUI 线程意外退出，子线程自动退出。
- **互斥锁 / 唯一实例**：使用 `win32event.CreateMutex` 确保同一程序只有一个实例运行，新实例启动时杀死旧实例。

### 4.4 配置与数据持久化

- **自动保存**：修改宏配置后，设置 3 秒防抖自动保存到 `macros.json`，而不是等到窗口关闭才保存。
- **备份机制**：保存前先备份旧文件（`macros.json.bak`），防止写坏。
- **配置校验**：加载时验证 JSON 结构完整性，损坏时自动从备份恢复。

### 4.5 调试与排障能力

- **日志系统**：将程序运行日志写入 `logs/` 目录（包含启动时间、热键注册状态、异常信息）。
- **健康检查 API**：运行一个本地 HTTP 接口（如 `http://127.0.0.1:18732/status`），返回 JSON 格式的运行状态。
- **进程列表导出**：提供导出当前运行配置和进程信息的功能，方便排查。

## 六、关键原则总结

1. **删除不先关 = 鬼进程** — 删除运行中的 Python 脚本前，一定要先通过 GUI 正常退出或 `Stop-Process` 杀掉进程。
2. **用户看不到 != 程序没在跑** — `pythonw.exe` 后台进程没有任何窗口，只能通过任务管理器或命令行发现。
3. **GUI 需要双向可见性** — 不仅有控制入口，还要有状态指示（系统托盘、颜色标签、日志）。
4. **紧急停止不能依赖 GUI** — 如果 GUI 被关闭但进程存活，需要独立的紧急停止机制（特定热键或命令行命令）。
5. **AI 项目之间的经验传递** — 这个文档可以直接交给另一个 AI，让它理解这个项目的上下文、架构优缺点和工程陷阱，无需额外解释。

## 七、Qt 全局热键线程经验（审查 #12）

### 问题与根因
`keyboard` 的全局热键回调运行在后台钩子线程。直接在回调中调用 Qt 控件、显示 OSD 或执行依赖 GUI 线程状态的逻辑，会造成跨线程访问风险；连续按 F9 还可能让多次执行重叠。

### 固定防范方式
钩子线程只能调用 `_HotkeyDispatcher.on_f9_hook` / `on_toggle_hook` 并发射 Qt 信号。Qt 主线程槽函数负责输入、OSD 和状态标签更新；F9 使用非阻塞 `threading.Lock`，未取得锁的连按直接丢弃，并在成功、异常和信号转发失败路径释放锁。

### 编码记录
`SETUP_GUIDE.md` 必须保留 UTF-8 BOM（`EF BB BF`），以便中文 Windows PowerShell 自动正确识别中文编码；更新文件后必须检查前三个字节。

## 八、热键重复事件与产品语义必须分开（审查 #13）

### 问题
阶段 1 的 `PRESS` 热键在按住 F9 时接收重复按下事件，导致 Hello World 在一次长按内反复执行。非阻塞锁只能避免重叠，不能把一次物理长按识别为一次触发。

### 防范原则
先明确产品语义，再选择实现：单次执行必须做“物理按下去重，释放后才允许下一次”；按住执行必须显式监听按下和释放，使用可取消的执行循环，并在释放时通知停止。不能将重复事件误当作已经实现的按住模式。

## 九、阶段边界必须以最新用户确认覆盖旧规格（审查 #15）

阶段 1 为验收而存在的 `PRESS` 不能被误写成阶段 2 的第四种模式；Quickinput 宏实际只参考 `switch`、`down`、`up`，单次由循环次数 `count: 1` 表达。阶段 2 的测试宏必须是 SendInput 的现实按键按下/释放，不能继续使用 Unicode `type_string` 伪装成游戏输入。

旧的阶段 2 草案含有 JSON 热键、`text` 步骤、`0–9999` 次和 `0.1–10×` 速度；已由用户确认修正为固定 F9/F12、`key_tap`、`0–99` 次和 `0.01–8×`。发现旧文档与最新确认冲突时，先更新规格和日志，再实现代码。

## 十、按下执行的停止条件必须同时参考模式和次数（审查 #17）

Quickinput 的 `down` 不是无条件“松开即停止”：按键释放只在 `count == 0` 的无限循环时调用停止；有限次数必须继续完成。因此阶段 2 只保留 `switch` 与 `down`，删除 `up` 和项目内部遗留的 `PRESS`。不能把一次手动验收的按住行为推广为所有次数配置的停止规则。

---

## 十一、外部配置的确认点要映射为下一次安全触发（审查 #18）

Quickinput 在用户按 Enter 确认后，立即更新内存中的 `macro.count`，下一次 `run_start` 使用新值。阶段 2 没有配置 UI，用户通过编辑器 Ctrl+S 保存 JSON；因此每次 F9 启动前必须严格重载文件，不能要求重启程序。运行中的序列仍使用启动快照，避免半路改次数造成不可预测行为；无效 JSON 必须拒绝启动且保留最后有效配置。

---

## 十二、快速保存 Python 宏不能依赖 import 缓存（审查 #19）

Python 的标准导入缓存可能在同一秒内复用“内容长度相同”的旧字节码，导致 `COUNT = 1` 改为 `COUNT = 0` 后下一次触发仍使用旧值。可信本地宏必须在每次触发前直接读取、`compile` 并 `exec` 已保存的源码；这与 Quickinput 的“确认后更新内存、下一次触发使用新配置”行为一致。

## 十三、v021 UI 外壳的成功验收方法（审查 #25-#26）

### 成功做法

1. **先冻结运行边界，再做外观。** UI 只在 `MainWindow._setup_ui()` 内实现；F9、F12、播放器、OSD、退出清理和真实输入路径均不改动，F2 只能显示为占位。
2. **视觉以“案例源码 + 项目参考图 + 截图”三方互证。** Quickinput 的 Candy 粉红主题源码锁定颜色、字体和控件层级；项目内四张图用于人工对照；离屏截图用于确认页面不是空白或布局崩坏。截图不能替代源码或用户的 Windows 验收。
3. **先自动验证安全，再让用户测试真实输入。** UI 测试用 offscreen `QApplication` 构造窗口外壳；热键注册和退出清理用 mock 验证，绝不在自动测试中发送真实键鼠输入。
4. **人工验收要分开记录。** 用户可以分别确认“热键与安全回归通过”“颜色方向满意”“视觉设计仍需调整”。不能因为 F9/F12 正常就宣称完整 UI 已最终验收。

## 十四、发布频率由人类验收门控制（审查 #30）

1. **默认不提交、不推送。** 日常实现、文档草稿和自动测试通过只说明本地工作可继续，不等于应创建 Git 版本或发布 GitHub。
2. **以当前版本的人类验收为发布门。** 只有用户明确通过当前版本的人类验收后，才可创建提交并发布；用户单独要求发布时才可例外。
3. **验收后的可见改动仍需确认。** 即使主体 UI 已通过，随后新增的中文文案等可见改动也应让用户做一次最小确认，再将该最终版本发布。
4. **发布前永远重验远端。** 先检查工作区、候选 SHA、远端默认分支和快进关系；只普通推送功能分支后再快进 `master`，并核验两个分支 SHA。网络异常、保护规则或非快进均立即停止，绝不使用 `--force`。
5. **案例优先必须落到可验证规则。** 用户提出的窗口高度想法与优秀案例 1 的最小高度 `510` 冲突时，因用户明确指定案例优先，保留 510；下一轮若要调整，必须先重新查证案例并取得新的 READY。

### 可复用的测试闭环

`RequirementCertifier READY` → UI 实现 → `unittest + compileall + diff check` → 文件长度检查 → 离屏截图 → 用户在 Windows 记事本验收 → Handover。

这个闭环在 v021 中让用户对运行、安全和粉红主题达到 90% 以上满意度，同时把“整体视觉仍需优化”保留为下一阶段的明确输入，而不是用主观猜测继续编码。

## 十五、GitHub 普通快进发布闭环（v021.1）

### 已验证的事实

- `git push` 使用仓库已保存的 Git HTTPS 凭据；没有安装 `gh`、或本轮 GitHub 插件没有暴露连接器写入接口，都不妨碍普通 Git 推送。
- 2026-07-16 已验证：`bc06485` 先发布到 `codex/v021-ui-shell`，再以普通快进发布到默认分支 `master`；远端 `HEAD`、两个分支最终均指向同一 SHA。
- 一次网络超时不能直接诊断为认证、插件授权或提交问题。先用 `git ls-remote --heads origin` 取得远端证据，再决定是否重试。

### 后续发布的固定步骤

1. 用 `git ls-remote --heads origin` 确认可访问远端；随后执行 `git fetch origin --prune`。
2. 确认工作区干净、`HEAD` 是预期候选提交，并用 `git merge-base --is-ancestor origin/master HEAD` 确认只会快进。
3. 先推送功能分支：`git push origin HEAD:<功能分支>`；再推送默认分支：`git push origin HEAD:master`。
4. 用 `git ls-remote --symref origin HEAD refs/heads/master refs/heads/<功能分支>` 核验默认分支、功能分支和远端 `HEAD` 指向预期 SHA。

### 失败处理与禁止项

- 连通性失败时不猜测原因；保留本地提交，先重新执行第一步。只有远端连通后才能重做完整预检。
- 遇到分支保护、非快进或远端在预检后变化，立即停止并重新审查；禁止 `--force`、rebase 或覆盖已发布历史。
- 只推送已通过相应需求门槛和验证的提交；优秀案例源码、图片与资产始终不进入提交范围。

## 九、阶段范围必须覆盖旧的长期规划（审查 #14）

### 问题
旧 `PROJECT_SPEC.md`、`PROJECT_TASKS.md` 和阶段测试计划同时写有 Python 脚本、编辑器、多脚本并发、组合键、窗口匹配与音效；交接要求的阶段 2 却只允许一个 JSON Hello World 宏、可中断播放器和四项配置。旧知识示例还把速度倍率写成等待时间相乘，和测试计划的“速度翻倍、延迟减半”相反。

### 防范原则
每个开发阶段都要有单独、可验证的最小规格，并明确它覆盖哪些长期愿景。实施前必须锁定 JSON 字段、模式、次数、速度和停止语义；速度一律按 `delay / speed` 计算。范围不清时只完善文档，不写代码。
